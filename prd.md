## 1. 产品概述

### 1.1 产品背景

在软件开发团队中，迭代需求管理涉及多个环节：规划会、进度跟踪、发布状态更新等。团队成员需要及时了解各阶段的关键信息，以确保项目按时交付。

目前，这些信息的传递主要依赖人工操作，存在以下问题：
- 容易遗漏重要通知
- 信息传递不及时
- 重复性工作消耗人力

### 1.2 产品定位

**迭代需求群消息自动通知系统** 是一个基于发版日历的自动化消息推送平台，通过集成云之家群机器人，实现迭代需求相关提醒消息的自动推送。

### 1.3 核心价值

| 价值点 | 说明 |
|--------|------|
| **自动化** | 根据发版日历自动推送消息，无需人工干预 |
| **智能化** | 支持节假日顺延、日期计算、禅道编号计算 |
| **可配置** | 消息模板、推送时间、成员配置均可自定义 |
| **可追溯** | 完整的推送日志记录，区分手动/自动推送，支持历史查询 |
| **本地优先** | 内置 SQLite 兼容模式，无需外部数据库即可本地运行 |

### 1.4 目标用户

- 项目经理
- 产品经理
- 开发团队负责人
- QA 负责人

### 1.5 部署模式

| 模式 | 数据库 | 适用场景 |
|------|--------|----------|
| **本地模式** | SQLite（内嵌） | 个人开发/测试，零配置启动 |
| **生产模式** | Supabase (PostgreSQL) | 团队协作，多实例部署 |

> 两种模式接口完全兼容，通过环境变量切换。本地模式自动初始化表结构和种子数据。

## 2. 系统架构

### 2.1 技术栈

| 层级 | 技术选型 |
|------|----------|
| 前端 | Next.js 15 + React 19 + TypeScript + Tailwind CSS |
| 后端 | Next.js API Routes |
| 定时任务调度 | 独立 Node.js cron worker 进程（node-cron），与 Next.js 应用分离部署 |
| 数据库 | SQLite (本地) / Supabase PostgreSQL (生产) |
| 消息推送 | 云之家群机器人 API |
| 认证 | API Key 认证（管理接口使用 `X-API-Key` Header，触发接口使用独立 `X-Trigger-Key`） |
| 时区 | 全局统一 Asia/Shanghai (UTC+8) |

> **双重数据库支持**：本地模式下使用 better-sqlite3 内嵌数据库，生产模式使用 Supabase。通过自定义 QueryBuilder 兼容层保持接口一致。

## 3. 功能模块

### 3.1 发版日历管理

#### 3.1.1 功能描述

管理全年发版日历，支持 CSV 格式导入，为消息推送提供基础数据。

#### 3.1.2 数据模型

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| year | INTEGER | 年份 |
| month | INTEGER | 月份 |
| iteration_number | TEXT | 迭代编号（如 0526） |
| planning_date | DATE | 规划会日期 |
| release_date | DATE | 发版日日期 |
| created_at | DATETIME | 创建时间（自动） |
| updated_at | DATETIME | 更新时间（自动） |

> **设计说明**：
> - 保留 `year`、`month` 字段便于按年月筛选和统计
> - 日期字段使用 DATE 类型，支持直接日期计算
> - 迭代周期定义为 `planning_date` ~ `release_date`，用于判定"当前迭代"
> - **注意**：规划会日期可能在前一个月甚至前一年（如 0106 迭代的规划会在 12/19）

#### 3.1.3 当前迭代判定规则

系统通过以下规则自动判定"当前迭代"：

```
当前迭代 = 满足以下条件的迭代记录：
  planning_date <= 当前日期 <= release_date

若存在多个匹配（数据异常），取 planning_date 最晚的那个。
若无匹配迭代，取 release_date 已过期但最近的一个迭代（降级策略）。
```

#### 3.1.4 CSV 导入格式规范

CSV 文件编码为 **GBK**，包含以下列头：

| 列头 | 示例 | 说明 |
|------|------|------|
| 年份 | 2026 | 迭代所属年份 |
| 月份 | 7 | 迭代所属月份 |
| 迭代编号 | 0707 | 迭代编号 |
| 发版日期 | 0707 | 发版日（M/D 或 MMDD 格式） |
| 规划会日期 | 6/18 | 规划会日期（可能跨月/跨年） |

**导入逻辑**：
- 使用 `iconv-lite` 解码 GBK → UTF-8
- 支持双引号括起的 CSV 字段
- 文件大小限制 ≤ 1MB
- 跨年规划会日期自动处理（如 0106 迭代的 12/19 → 2025-12-19）
- 导入前校验：迭代编号不能重复

#### 3.2.1 功能描述

#### 3.2.2 数据模型

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| name | TEXT | 机器人名称 |
| webhook_url | TEXT | Webhook 地址 |

### 3.3 消息模板配置

#### 3.3.1 功能描述

配置各类消息的模板内容，支持丰富的占位符计算。

#### 3.3.2 消息类型枚举

| 类型值 | 名称 | 说明 | 默认推送时间 |
|--------|------|------|--------------|
| 1 | 规划会提醒 | 提醒团队进行需求规划 | 10:00 |
| 2 | 进度更新提醒 | 提醒更新需求进度 | 11:00 |
| 3 | 发版后状态更新提醒 | 提醒更新发布状态 | 09:15 |

#### 3.3.3 占位符支持

| 占位符 | 示例值 | 说明 |
|--------|--------|------|
| `{发版日日期}` | 7/7 | 发版日日期（M/D） |
| `{发版日日期+N}` | 7/10 | 发版日后N天 |
| `{发版日日期-N}` | 7/6 | 发版日前N天 |
| `{发版日日期所在星期}` | 周二 | 发版日是周几 |
| `{规划会日期}` | 6/18 | 规划会日期 |
| `{规划会日期+N}` | 6/19 | 规划会后N天 |
| `{规划会日期-N}` | 6/17 | 规划会前N天 |
| `{规划会日期所在星期}` | 周四 | 规划会是周几 |
| `{禅道编号}` | 351 | 当前迭代禅道编号 |
| `{禅道编号+N}` | 353 | 编号+N |
| `{禅道编号-N}` | 350 | 编号-N |

> **替换顺序**：先处理带偏移量的（如 `{发版日日期+3}`），再处理无偏移量的（如 `{发版日日期}`），防止前缀匹配误伤。

#### 3.3.4 禅道编号计算

```
所有迭代按 planning_date ASC 排序 →
  baseIndex = 基准迭代在排序列表中的位置
  currentIndex = 当前迭代在排序列表中的位置
  禅道编号 = base_chanzhou_num + (currentIndex - baseIndex) × increment
```

### 3.4 推送时间配置

#### 3.4.1 功能描述

配置各类消息的推送时间，支持节假日自动顺延。每种消息类型绑定一个机器人。

#### 3.4.2 手动推送

每条消息类型卡片底部提供「手动推送」按钮，点击后立即渲染模板并推送到绑定机器人群聊，无需等待定时触发。推送日志标记 `push_source = "manual"`。

---

### 3.5 @成员配置

#### 3.5.1 功能描述

配置各类消息需要 @ 的成员，支持按消息类型分别配置。

#### 3.5.2 简化设计

用户**只需输入成员姓名**，无需填写 UserID。系统自动将 `userid` 设为与 `member_name` 相同值。消息推送时使用 `@姓名` 格式。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| message_type | INTEGER | 消息类型 |
| member_name | TEXT | 成员姓名 |
| userid | TEXT | 用户标识（自动同步 member_name） |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

> **编辑同步**：编辑成员姓名时，userid 自动同步更新，确保推送 @ 使用最新名称。

---

### 3.6 推送日志

#### 3.6.1 数据模型

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| message_type | INTEGER | 消息类型 |
| iteration_number | TEXT | 推送的迭代编号 |
| content | TEXT | 完整推送内容 |
| status | TEXT | success / failed / preview / retrying |
| push_source | TEXT | auto（自动）/ manual（手动） |
| retry_count | INTEGER | 已重试次数 |
| error_message | TEXT | 错误信息 |
| pushed_at | DATETIME | 计划推送时间 |
| completed_at | DATETIME | 实际完成时间 |
| created_at | DATETIME | 创建时间 |

#### 3.6.2 推送来源区分

| 来源 | push_source | 触发方式 |
|------|-------------|----------|
| 自动推送 | auto | Cron Worker 定时触发 |
| 手动推送 | manual | 管理页面「手动推送」按钮 |

## 4. 启动方式

### 本地模式（推荐）

```bash
# 1. 安装依赖
npm install

# 2. 创建配置文件
cp .env.example .env.local

# 3. 启动 Web 管理页面 (端口 3001)
npm run dev

# 4. 启动 Cron Worker（可选）
npm run worker
```

数据库自动创建在 `data/local.db`，含预置种子数据（26 条日历 + 7 个节假日 + 3 个模板 + 机器人配置）。
