### 1.3 双数据库模式

系统通过 `src/lib/supabase.ts` 实现 Supabase SDK 兼容层：

```
┌──────────────────────────────────────────────┐
│              业务代码                         │
│  supabaseAdmin.from("table").select()        │
│  supabaseAdmin.from("table").insert()        │
│  supabaseAdmin.from("table").update()        │
│  supabaseAdmin.from("table").delete()        │
├──────────────────────────────────────────────┤
│          QueryBuilder 兼容层                  │
│  .eq() .neq() .gt() .gte() .lt() .lte()     │
│  .order() .limit() .range() .select()        │
│  .single() .insert() .update() .delete()    │
├──────────────────────────────────────────────┤
│  后端: better-sqlite3 (本地/SQLite)          │
│  Web: Supabase SDK (生产/PostgreSQL)         │
└──────────────────────────────────────────────┘
```

**关键转换**：
- `true`/`false` → `1`/`0`（SQLite 不支持 boolean）
- `IS NULL` / `IS NOT NULL` 处理
- `single()` 兼容数组和单对象返回
- `select()` 不覆盖写入操作（INSERT/UPDATE/DELETE 保留原有 op）

- **x-api-key**：管理类操作（CRUD），用于前端管理页面调用

## 3. 核心业务流程

### 3.1 消息推送完整流程

```
Cron Worker (定时触发) 或 手动推送按钮
       │
       ▼
  ┌──────────────────────────────────────────────────────┐
  │ 1. 检查工作日                                        │
  │    - 周末 (周六/周日) → 跳过（手动推送不检查）        │
  │    - 节假日 (holidays表匹配) → 跳过                  │
  └──────────┬───────────────────────────────────────────┘
             │ 是工作日
             ▼
  ┌──────────────────────────────────────────────────────┐
  │ 2. POST /api/trigger                                │
  │    Headers: x-trigger-key                           │
  │    Body: { message_type, push_source?, iteration_number? }
  └──────────┬───────────────────────────────────────────┘
             │
             ▼
  ┌──────────────────────────────────────────────────────┐
  │ 3. /api/trigger 处理流程                             │
  │                                                      │
  │  3a. 从 push_time_config 获取 robot_id               │
  │  3b. 从 robot_config 获取 webhook_url                │
  │  3c. 确定推送迭代：                                   │
  │      类型1 (规划会提醒) → 下一个迭代 (planning_date >= today)
  │      类型2 (进度更新)   → 当前迭代                    │
  │      类型3 (发版后)     → 当前迭代 (release_date + 1) |
  │  3d. 从 message_templates 获取模板                    │
  │  3e. 计算禅道编号                                     │
  │  3f. 替换占位符                                       │
  │  3g. 附加 @userid（从 members 表）                    │
  │  3h. POST JSON → 云之家 Webhook                       │
  │  3i. 记录 push_logs（含 push_source）                  │
  └──────────────────────────────────────────────────────┘
```

### 3.2 规划会提醒 - 迭代选择逻辑

> **关键区别**：规划会提醒使用**下一个迭代**（planning_date >= today 的第一个），而非当前迭代。

```
类型1 (规划会提醒):
  查询 release_calendar WHERE planning_date >= today
  按 planning_date ASC 取第一条
  → 这是即将召开规划会的下一个迭代
  → 若当前日期 > 当前迭代的 planning_date，不报当前迭代，报下一个

类型2 (进度更新提醒):
  当前迭代: planning_date <= today <= release_date

类型3 (发版后状态更新):
  当前迭代: planning_date <= today <= release_date
  基准日期 = release_date + 1天
```

### 3.4 占位符替换规则

**替换顺序**：先处理带偏移量的（如 `{发版日日期+3}`），再处理无偏移量的，防止前缀匹配误伤。

| 占位符 | 示例输入 | 示例输出 | 说明 |
|--------|----------|----------|------|
| `{禅道编号}` | `{禅道编号}` | `351` | 当前禅道编号 |
| `{禅道编号+N}` | `{禅道编号+2}` | `353` | 偏移 |
| `{禅道编号-N}` | `{禅道编号-1}` | `350` | 偏移 |
| `{发版日日期}` | `{发版日日期}` | `7/7` | M/D 格式 |
| `{发版日日期+N}` | `{发版日日期+3}` | `7/10` | 后N天 |
| `{发版日日期-N}` | `{发版日日期-1}` | `7/6` | 前N天 |
| `{发版日日期所在星期}` | `{发版日日期所在星期}` | `周二` | 中文星期 |
| `{规划会日期}` | `{规划会日期}` | `6/18` | M/D 格式 |
| `{规划会日期+N}` | `{规划会日期+1}` | `6/19` | 后N天 |
| `{规划会日期-N}` | `{规划会日期-1}` | `6/17` | 前N天 |
| `{规划会日期所在星期}` | `{规划会日期所在星期}` | `周四` | 中文星期 |

### 3.5 下次推送时间计算

```
对于每种 message_type (1/2/3)：
  │
  ├─ 类型1 (规划会提醒):
  │   选取下一个迭代: planning_date >= today, ASC 第一条
  │   基准日期 = 该迭代的 planning_date
  │
  ├─ 类型2 (进度更新提醒):
  │   当前迭代: planning_date <= today <= release_date
  │   基准日期 = today
  │
  ├─ 类型3 (发版后状态更新):
  │   当前迭代: planning_date <= today <= release_date
  │   基准日期 = release_date + 1天
  │
  ├─ 工作日调整:
  │   若为周末 → 顺延到下周一
  │   若为节假日 → 顺延到假期结束后第一个工作日
  │   (最多推进30天防死循环)
  │
  └─ 组合: 调整后日期 + push_time_config.hour:minute
      → 返回字段: next_time, iteration (前端用)
```

### 3.6 CSV 导入流程

```
1. 接收 FormData (file 字段)
2. 校验文件大小 ≤ 1MB
3. iconv-lite 解码 GBK → UTF-8
4. 逐行解析 (支持双引号括起的 CSV 字段)：
   年份, 月份, 迭代编号, 发版日期, 规划会日期
5. 日期解析 (parseDateValue)：
   - M/D → 如 "12/19" → 12月19日
   - MMDD → 如 "0106" → 1月6日
   - 跨年: 月份 > 迭代月份 → 年份-1
6. 去重 (同批次 + 数据库)
7. 批量 INSERT
8. 返回: { imported: N, errors: [...] }
```

### 3.7 推送失败重试机制

```
Cron Worker 每 30 分钟扫描:
  push_logs WHERE status IN ('failed', 'retrying')
    AND retry_count < 3

  重试: 重新调用云之家 Webhook
  成功 → status = 'success', completed_at = now
  失败 → retry_count += 1
    retry_count < 3 → status = 'retrying'
    retry_count >= 3 → status = 'failed'
```

### 3.8 防御性编程

| 场景 | 处理 |
|------|------|
| 模板内容为空 | 返回 400 "模板内容为空" |
| 禅道编号为负数 | 防御为 1 |
| 分页过大 | pageSize 上限 200 |
| 时区 | 全局 utcToZonedTime Asia/Shanghai |
| NaN 检查 | retry_count 非数字时默认为 0 |
| 分布式锁 | 本地 SQLite 模式注释掉（单实例够用） |

## 6. Cron Worker 设计

### 6.1 进程模型

Worker 是独立 Node.js 进程，通过 `npx tsx src/worker/index.ts` 启动。

```
┌──────────────────────────────────────────┐
│              Cron Worker                  │
├──────────────────────────────────────────┤
│  启动流程：                               │
│  1. dotenv 加载 .env.local               │
│  2. 环境变量校验 (API_KEY, TRIGGER_KEY)  │
│  3. 连接数据库 (通过 supabase.ts)        │
│  4. loadAndSchedule() 注册 cron jobs     │
│  5. 启动心跳定时器 (5min)                │
│  6. 启动重试扫描器 (30min)               │
│  7. 启动配置热加载 (10min)               │
├──────────────────────────────────────────┤
│  优雅退出：                               │
│  SIGINT / SIGTERM → 停止所有 cron → exit │
└──────────────────────────────────────────┘
```

### 6.2 定时任务

| 任务 | 周期 | 说明 |
|------|------|------|
| 主推送 | cron `{min} {hour} * * *` | 每种消息类型独立 |
| 心跳 | 每 5 分钟 | worker_heartbeat |
| 重试扫描 | 每 30 分钟 | retrying + failed + retry_count<3 |
| 配置热加载 | 每 10 分钟 | push_time_config 变更 |
